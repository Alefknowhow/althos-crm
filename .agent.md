# Projeto: Althos CRM

## Sobre
CRM multi-tenant para agências de marketing e clínicas. Substitui Hubspot + ClickUp + Typeform + ManyChat + ActiveCampaign em uma única plataforma. Construído pela Althos Performance (agência do Alef Trentin, Itajaí-SC).

## Stack obrigatória
- Next.js 14 (App Router) + TypeScript estrito (`strict: true`)
- Tailwind CSS + shadcn/ui (tema "neutral", style "new-york")
- Supabase: Postgres + Auth + Storage + Row Level Security
- Server Actions para mutations, Server Components para reads
- Zod para validação de toda entrada externa
- Inngest para jobs em background e automações
- Resend para envio de e-mail
- Meta WhatsApp Cloud API (oficial)
- Stripe (ou Pagar.me) para billing
- Deploy: Vercel
- Monitoramento: Sentry

## Regras inegociáveis

### Segurança e multi-tenancy
1. Toda tabela com dados de cliente TEM coluna `organization_id` e policies de RLS configuradas para SELECT, INSERT, UPDATE, DELETE.
2. NUNCA use `SUPABASE_SERVICE_ROLE_KEY` no client-side. Service role só em Server Actions específicas, auditadas, e documentadas.
3. Padrão de RLS: usuário acessa registros cuja `organization_id` está em `(SELECT organization_id FROM memberships WHERE user_id = auth.uid())`.
4. Toda entrada externa (forms, API routes, webhooks, server actions) é validada com Zod ANTES de qualquer lógica.
5. Webhooks validam assinatura criptográfica antes de processar payload (Resend, Stripe, Meta).

### Qualidade de código
6. Tipagem forte em tudo. Tipos do Supabase gerados via `npx supabase gen types typescript --project-id XXX > types/supabase.ts`. Nunca use `any` — se precisar de flexibilidade, use `unknown` e valide com Zod.
7. Toda Server Action retorna `{ ok: true; data: T } | { ok: false; error: string }`. Nunca lança exceção sem tratar.
8. Componentes UI vêm de shadcn/ui. Não criar componente novo se já existe equivalente (Button, Dialog, Sheet, Form, Input, Select, etc).
9. Comentários explicam o "porquê", nunca o "o quê". Código legível dispensa comentário descritivo.
10. Sem `console.log` em código de produção. Use logger estruturado (Pino ou similar) com níveis info/warn/error.
11. Toda alteração de schema do banco passa por migration versionada em `/supabase/migrations/NNNN_descricao.sql`.

### UX e performance
12. Mobile-first em toda interface pública (formulários, landing).
13. Loading states com skeleton do shadcn em toda tela com fetch.
14. Empty states informativos com CTA em toda lista vazia.
15. Toasts (Sonner do shadcn) para feedback de ações.
16. Optimistic updates onde fizer sentido (drag-drop, toggle, marcar como concluído).

## Estrutura de pastas

```
/app
  /(public)              # rotas públicas: landing, login, signup
  /(public)/f/[slug]     # forms públicos (sem auth)
  /api/webhooks          # webhooks (Stripe, Resend, Meta, Inngest)
  /app/[orgSlug]         # área autenticada multi-tenant
  /super-admin           # área restrita ao super admin
/components
  /ui                    # componentes shadcn (não editar manualmente)
  /features              # componentes do domínio (LeadCard, KanbanColumn, etc)
/lib
  /supabase              # clientes server.ts, client.ts, middleware.ts
  /inngest               # client e definições de funções
  /validators            # schemas Zod compartilhados
  /whatsapp              # cliente Meta Cloud API
  /utils                 # helpers genéricos
/actions                 # Server Actions agrupadas por domínio (leads.ts, tasks.ts, etc)
/types                   # types gerados do Supabase + types do domínio
/supabase
  /migrations            # SQL versionado
/scripts                 # scripts úteis (test-rls, seed, etc)
/public                  # assets estáticos
```

## Hierarquia de dados

```
auth.users (Supabase Auth)
  └── memberships (relação N:N)
        └── organizations
              ├── pipelines → pipeline_stages
              ├── leads → lead_activities
              ├── tasks
              ├── forms → form_submissions
              ├── whatsapp_conversations → whatsapp_messages
              ├── email_templates → email_sends
              └── automations → automation_runs
```

## Roles dentro da organização
- `owner`: acesso total + billing
- `admin`: acesso total exceto billing
- `member`: pode trabalhar com leads/tasks, sem editar configurações

## Super admin (eu, Alef)
Usuários com `auth.users.raw_user_meta_data->>is_super_admin = 'true'` podem:
- Acessar `/super-admin` (lista de orgs)
- Impersonar qualquer organização (com banner amarelo persistente e auditoria)
- Em modo impersonation: leitura + edições suaves, NÃO pode deletar dados nem mudar credenciais críticas

## Convenções de nomenclatura
- Tabelas: snake_case plural (`leads`, `lead_activities`)
- Colunas: snake_case (`created_at`, `organization_id`)
- Componentes React: PascalCase (`LeadCard`, `KanbanBoard`)
- Hooks: camelCase com prefixo `use` (`useCurrentOrg`)
- Server Actions: camelCase com verbo (`createLead`, `updateLeadStage`)
- Types: PascalCase (`Lead`, `LeadInput`, `LeadActivityType`)
- Constantes: SCREAMING_SNAKE_CASE (`MAX_LEADS_STARTER`)

## Tratamento de erros
- API routes: retornam JSON `{ error: string, code: string }` com status HTTP correto.
- Server Actions: nunca throw. Sempre `{ ok: false, error: string }`.
- Erros de banco (constraint violation, etc): traduzir pra mensagem em português pro usuário.
- Erros inesperados: logar no Sentry + retornar mensagem genérica "Algo deu errado. Tente novamente."

## Testes obrigatórios
- Toda feature crítica (auth, RLS, billing, automações) tem teste manual documentado.
- Script `/scripts/test-rls.ts` roda em todo deploy de schema novo.
- Smoke test E2E (Playwright) do fluxo principal: signup → criar form → submeter form → ver lead → criar tarefa.

## Princípio diretor
Velocidade SEM dívida técnica perigosa. Estamos em corrida de 30 dias pra MVP, mas multi-tenant com RLS errada vaza dados de cliente — isso destrói o negócio. Quando houver trade-off entre velocidade e segurança/correção do banco, escolha SEMPRE a opção segura, mesmo que custe meio dia a mais.
